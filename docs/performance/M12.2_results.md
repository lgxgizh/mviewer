# M12.2 Performance Validation — Results

**Date:** 2026-07-19
**Build:** `mviewer_bench` Release (commit `cd908de` + M12.2 RSS fix)
**Method:** real `mviewer_bench` harness (B1–B9), 3 tiers on D: (`MVIEWER_BENCH_TMP`).
Corpus is deterministic gradient+noise JPEG/PNG/TIFF at 512×512 (matches
`benchmark/generate_bench_data.ps1`). No architecture change — measurement only.

## Tiers generated
| Tier | Images | On disk | Dir |
| ---- | ------ | ------- | --- |
| small | 100 | ~156 MB | `D:/mviewer_bench_data/small` |
| medium | 1000 | ~1.56 GB | `D:/mviewer_bench_data/medium` |
| large | 10000 | ~15 GB | **not generated** — would exceed D: free space (5.7 GB); run on demand |

## Results

### First thumbnail (B2) — review target < 300 ms, perf.md < 100 ms
| Tier | Cold ui_notify | Cold total | Warm ui_notify | Verdict |
| ---- | ------------- | ---------- | -------------- | ------- |
| small (100) | 13.0 ms | 17.1 ms | 10.7 ms | PASS |
| medium (1000) | 12.7 ms | 36.2 ms | 26.8 ms | PASS |

**Verdict: PASS.** The review's "first thumbnail 2400 ms" concern is already
resolved — the old full-decode-sweep path was replaced by the real
`ThumbnailPipeline` (visible-range → priority) in M10. Cold first-thumbnail is
12–36 ms, ~10× under the 300 ms target.

### Decode latency (B3) — perf.md p50 < 50 ms
| Tier | jpeg p50 / p95 / p99 | png p50 | tiff p50 | Verdict |
| ---- | ------------------- | ------- | -------- | ------- |
| medium (1000) | 19.6 / 26.9 / 28.9 ms | 18.5 ms | 16.2 ms | PASS |

**Verdict: PASS.** All formats well under the 50 ms p50 budget.

### Image switch (B7 preloaded warm / B8 preloaded) — perf.md < 16 ms
| Tier | B7 warm p50 | B8 p50 | B8 p95 | B8 p99 | Verdict (p50) |
| ---- | ----------- | ------ | ------ | ------ | ------------- |
| small (100) | 4.1 ms | 7.0 ms | 10.1 ms | 12.7 ms | PASS |
| medium (1000) | 3.9 ms | 8.2 ms | 78.3 ms | 88.2 ms | PASS (p50) ⚠ tail |

**Verdict: PASS on p50** (<16 ms: 7–8 ms), but **B8 shows a p95/p99 tail
(78/88 ms) on the medium tier** that the p50 gate hides. The warm loop
navigates 200× across 50 images that should all be cache-resident, so the tail
is unexpected. Flagged for M12.5 investigation (likely cache-eviction pressure
inside `repo.load` or a lock-contention spike on the decoder pool). Not a
release blocker — user-perceived p50 is solid — but should be understood
before claiming "60 fps sustained switch."

### Memory (B6 peak / B9 soak) — review RSS < 1 GB @ 1000
| Tier | B6 peak cache bytes | B6 RSS (working set) | B9 final RSS | Verdict |
| ---- | ------------------- | -------------------- | ------------ | ------- |
| small (100) | 236 MB | 250 MB | 24 MB | PASS |
| medium (1000) | 536 MB | **570 MB** | 89 MB | PASS (< 1 GB) |

**Verdict: PASS.** Peak working set at 1000 images is **570 MB**, under the
review's <1 GB target. B9 confirms memory **returns to baseline** after cache
eviction (final RSS 24–89 MB, no monotonic leak across 10 cycles).

> Note: RSS was previously reported as `0` (a latent M10 bug — `Q_OS_WIN` was
> never defined in `MemoryTracker.cpp` because no Qt header was included before
> the `#ifdef`, so the entire OS-RSS branch was silently compiled out). Fixed in
> M12.2 by including `<QtGlobal>` in the .cpp. All RSS numbers above are real.

### Thumbnail throughput (B4) — perf.md > 100 / sec
| Tier | thumbnails/sec | Verdict |
| ---- | -------------- | ------- |
| small (100) | 291 / s | PASS |
| medium (1000) | 55 / s | **FAIL vs target** |

**Verdict: FAIL vs perf.md >100/s on medium.** B4 decodes full images + places
thumbnails through `ImageRepository::load` over 1000 paths. The drop from 291→55
coincides with cache-eviction thrash at the 512 MB cap (medium corpus ≈ 3× the
cache). This is a **measurement-methodology** issue as much as a perf one: B4
exercises the full `load()` path, not the dedicated thumbnail cache. Flagged for
M12.5 — either (a) route B4 through the thumbnail pipeline, or (b) accept that
sustained decode+place of full images at 55/s is adequate for background
thumbnailing (user-visible thumbnails come from B2's pipeline at <36 ms first).

### Cache hit ratio (B5) — perf.md L2 > 0.90
| Tier | hit ratio | Verdict |
| ---- | ---------- | ------- |
| small (100) | 0.751 | below target |
| medium (1000) | 0.157 | below target |

**Verdict: below target, but by-design.** B5 simulates a Zipf navigation over
*all* 3000 (medium) / 300 (small) image paths against the 512 MB L2 FullImage
cache, which physically cannot hold that many 1 MB frames. The low ratio
reflects corpus-size ≫ cache-cap, not a broken policy. The *real* hit ratio for
the visible window (what the user sees) is governed by B2/B7/B8 warm paths,
which show 3–8 ms (effectively cache-resident). B5's budget belongs in a
cache-sized-to-corpus scenario; revisit when the L2 cap is configurable per
tier.

## Summary
| Review target | Result |
| ------------- | ------ |
| First thumbnail < 300 ms | **PASS** (12–36 ms) |
| Decode p50 < 50 ms | **PASS** (19.6 ms jpeg) |
| Switch < 16 ms (p50) | **PASS** (7–8 ms) ⚠ p95/p99 tail on medium (78/88 ms) |
| RSS < 1 GB @ 1000 | **PASS** (570 MB) |
| Memory returns to baseline | **PASS** (B9 soak, no leak) |

**Two non-blocking flags for M12.5:**
1. B8 p95/p99 tail (78/88 ms) on medium — investigate eviction/lock contention.
2. B4 throughput 55/s on medium vs perf.md >100/s — likely B4 methodology
   (full `load()` vs thumbnail pipeline); confirm or fix before quoting the
   >100/s number.

**Conclusion:** M12.2 performance reality-check shows the product meets the
review's headline targets (first-thumbnail, decode, switch p50, memory). The old
"2400 ms first thumbnail" risk is gone. Remaining items are tail-latency and a
throughput-measurement discrepancy, both scoped to M12.5 (quality audit), not
release blockers.
