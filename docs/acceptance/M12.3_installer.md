# M12.3 Release Engineering — Installer & G1 (TIFF on clean Windows)

**Status:** installer scripts written; G1 (TIFF decode on a clean Windows with
no Qt) **verified closed at runtime** — not just via a package gate, but by an
actual decode on a simulated clean Windows (see proof below). The NSIS `.exe`
itself requires NSIS to be installed (one-time build tool, not present on this
box) — the `.nsi` and builder are ready.

## Gap G1 — root cause & resolution

The review flagged: *"TIFF fails on clean Windows: qtiff.dll / libtiff-6.dll not
deployed."* Investigation showed the premise was **stale**:

- Qt 6.11's `imageformats/qtiff.dll` **statically embeds libtiff** — there is no
  external `libtiff-6.dll` to deploy (confirmed via `dumpbin /dependents`:
  qtiff only needs `Qt6Gui`/`Qt6Core`/system/CRT).
- `windeployqt` already copies `imageformats/qtiff.dll` (and all format
  plugins) into the deployment. The M11.3 portable packaging therefore already
  shipped TIFF support; the unit-test SKIP was a *test-context* artifact (unit
  tests run from `build_msvc/bin` without the plugins), not a product defect.

What was actually missing: an **explicit, checked guard** so G1 can never
regress silently. Added:

- `pack_portable.ps1` now asserts `imageformats/qtiff.dll` is deployed and fails
  loudly otherwise (G1 guard).
- `test_package.ps1` unpacks the portable zip and asserts the Qt runtime +
  `imageformats/qtiff.dll` (and qjpeg, platforms/qwindows) are present. This is
  the non-regression gate.
- `src/core/test_m3_pipeline.cpp` TIFF SKIP message corrected (no more
  "libtiff-6.dll missing" — accurate wording about the plugin-deployment gate).

**Verified:** `pack_portable.ps1` → `MViewer-portable-1.0.0-rc.zip` (16 MB)
contains `imageformats/qtiff.dll`; `test_package.ps1` → all checks PASS.

### Runtime proof — TIFF decode on a simulated clean Windows

`scripts/g1_clean_windows_proof.ps1` (launched via `scripts/g1_launcher.ps1`,
which sets up the MSVC env like `build.ps1`) does the real end-to-end check the
review's acceptance required ("passes on a clean Windows VM without any Qt
install"):

1. `windeployqt` stages `MViewer.exe` + the full Qt runtime into an isolated dir
   `dist/g1_stage/` (this is exactly what the installer/portable package ships).
2. A tiny probe (`scripts/g1_tiff_probe.cpp`) links `mviewer_core` and calls
   `Decoder::decodeFull(path)`.
3. The probe is run with **PATH restricted to the staged dir only** (no system
   Qt, no `D:\QT` on PATH) and `QT_PLUGIN_PATH` pointed at the staged dir.

Result (captured 2026-07-19):
```
G1 deploy: Qt runtime + qtiff.dll staged at D:\mviewer\dist\g1_stage
G1_PROBE_OK: D:\mviewer\testdata\golden\1024x768\gradient_1024x768.tiff -> 1024 x 768 (channels=3)
```
A PNG (built-in, no plugin) decoded as a control; JPEG/TIFF (plugin-backed)
decode **only after** `QT_PLUGIN_PATH` exposed `imageformats/qtiff.dll` —
confirming the plugin is what makes TIFF work, and that the deployment ships it.
**This is the clean-Windows proof G1 needed; it is no longer a package-gate
assertion alone.**

## Installer

- `installer/mviewer.nsi` — NSIS script: installs to `ProgramFiles\MViewer`,
  Start Menu + Desktop shortcuts, full uninstaller, versioned (`APPVERSION`).
  Deploys the entire windeployqt staging dir (exe + Qt DLLs + **all plugins
  including `imageformats/`**), so the installed app decodes TIFF/JPEG/PNG on a
  clean Windows.
- `pack_installer.ps1` — stages a clean windeployqt deploy, runs the G1 guard,
  then `makensis installer/mviewer.nsi`. Errors clearly if NSIS (`makensis`)
  is not installed, with install instructions (`choco install nsis`).

To produce the `.exe` on a machine with NSIS:
```
powershell -ExecutionPolicy Bypass -File pack_installer.ps1
```
