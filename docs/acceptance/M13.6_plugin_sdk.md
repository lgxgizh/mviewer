# M13 Phase 6 — Plugin SDK stabilization (acceptance)

**Status:** DONE (verified). SDK documented, ABI contract nailed down, loader/
manager unit tests wired into the gate. The demo plugin builds, is loaded, and
is analyzed end-to-end by MViewer (ctest `pluginregistry_tests`).
**RFC:** `docs/rfc/M13_PRODUCT_BETA.md` Phase 6.
**Date:** 2026-07-19.

## Goal (review Phase 6)

> Third party can `git clone → compile → drop dll → recognized`.
> Document the existing `PluginLoader`/`PluginManager`/`example_analyzer` as the
> SDK; ship a minimal demo plugin build; clarify the ABI contract (ADR-005).

## What was done (stabilize + document — no new analyzer types)

1. **`docs/sdk/PLUGIN_SDK.md`** — the stable plugin contract: the `Analyzer`
   interface, the three frozen `extern "C"` exports (`createAnalyzer` /
   `destroyAnalyzer` / `pluginName`), the ABI stability rules (same toolchain,
   same Qt, `mviewer_core` must stay SHARED, vtable evolution policy, no
   runtime unload), in-tree + out-of-tree build recipes, and the load path.

2. **`plugins/example/README.md`** — how to build the reference plugin and use
   it as a template. The example (`ExampleAnalyzerPlugin.cpp`, mean-luminance)
   is the canonical demo.

3. **`docs/adr/005-why-plugin-analysis.md`** — resolved the "Plugin ABI
   stability across versions (future concern)" bullet into a concrete,
   enforced contract section; cross-linked to the SDK doc.

4. **ctest gating for loader/manager tests** — `test_plugin_loader` and
   `test_plugin_manager` were **built but never registered as ctest** (so they
   could bit-rot). Added `add_test(pluginloader_tests)` /
   `add_test(pluginmanager_tests)` in `src/CMakeLists.txt`. Both pass.

## Verification (evidence)

- `example_analyzer.dll` is built by the main build (`add_subdirectory(plugins/
  example)` in root `CMakeLists.txt`) and lands in `build_msvc/bin/`.
- `ctest pluginregistry_tests` → **PASS**: load → self-register with
  `AnalyzerRegistry` → `create()` → `analyze()` → `resultText()` →
  `analyzeRegion()` on a 10×10 `Selection`. This is the RFC acceptance
  (compiles against SDK headers + loaded by MViewer).
- `ctest pluginloader_tests` → PASS (scan / load-invalid / error message).
- `ctest pluginmanager_tests` → PASS (lifecycle / scan / unload-nonexistent).

## Known issue (NOT a Phase 6 regression — flagged, out of scope)

Under `build.ps1 Test` (`ctest -j4`), 7 pre-existing tests
(`core_tests`, `m3m4m5`, `m3pipeline`, `decoder_tests`, `cache_tests`,
`repository_tests`, `metadata_tests`) intermittently FAIL when run 4-at-a-time,
but **PASS 100% when run serially** (`ctest -j1`) and passed in the prior full
run. Root cause: shared singletons (`TaskScheduler`, `AnalyzerRegistry`,
`MemoryTracker`) and runtime-generated fixtures are not isolated across
parallel test processes — a pre-existing test-isolation smell, exposed more
often now that 2 more parallel workers exist. It is **not** caused by Phase 6
(the 2 new plugin tests pass under both serial and parallel; the 7 flaky tests
pass serially).

Fixing it means isolating shared state per-test or serializing those 7 — that
touches test infrastructure / singleton lifecycle, outside Phase 6 scope and
the frozen `build.ps1`/`ci.yml`. Tracked here for the next test-quality pass;
the Phase 6 deliverable itself is green.
