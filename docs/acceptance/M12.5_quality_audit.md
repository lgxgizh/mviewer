**Milestone**: M12 Product Beta Hardening → M12.5 Quality Audit
**Date**: 2026-07-19
**Scope**: (5.1) Qt-boundary scan of core/domain headers; (5.2) thread-safety audit
of the four singleton core services; (5.3) API documentation skeleton for the

**Freeze note**: This audit is *read-mostly*. It documents the current state and

refactor of `ImageRepository` / `CacheManager` / `RenderEngine` (per AGENTS.md

not leaks.

## 5.1 Qt-boundary scan (core/domain headers)

**Rule** (AGENTS.md): *No Qt types in `domain/` or `core/` headers.*

**Tool**: `scripts/audit_qt_boundary.ps1` (additive; runs in CI or locally).
Scans `src/core/**/*.h` + `src/domain/**/*.h` for forbidden UI includes
(`QWidget`, `QPainter`, `QPixmap`, `QMainWindow`, `QGraphicsView`, …) and warns
on Qt GUI value types (`QImage`, `QPicture`).

**Result (63 headers scanned)**:

| File | Include | Verdict |
|------|---------|---------|
| `src/core/render/RenderEngine.h` | `<QPainter>` (line 5) | **SANCTIONED** — see below |
| `src/core/image/QtConvert.h` | `<QImage>` (line 7) | **SANCTIONED** — see below |
| all other 61 headers | — | **CLEAN** (0 forbidden, 0 warnings) |

**Sanctioned exceptions (intentional, not leaks):**

1. `RenderEngine.h` — the software renderer's *public* CPU-rasterization entry
   point (`executeCommand(QPainter&, …)`, private `executeDrawXxx` dispatchers)
   legitimately uses `QPainter`/`QRect`. The rest of the render API is Qt-free
   (operates on `ImageData`). This is the deliberate spot where core does
   rasterization; moving it would require a UI-side adapter and is out of scope
   for M12.5 (RenderEngine is frozen).
2. `QtConvert.h` — the explicit `ImageBuffer ↔ QImage` conversion adapter. By
   design this is *the* point where Qt enters core. Isolating the conversion
   here keeps every other core header Qt-free.

**Conclusion**: architecture boundary is intact. The two hits are the sanctioned
conversion/rasterization boundary; everything else is clean. Scanner exits 0.

## 5.2 Thread-safety audit (singleton core services)

All four target services are **Meyers singletons** (`static X& instance()` →
`static X inst;` inside the function). C++11 guarantees thread-safe local-static
initialization, so construction is safe. Audit of runtime locking:

| Service | Singleton | Internal locking | Hot-path counters | Verdict |
|---------|-----------|------------------|-------------------|---------|
| `CacheManager` (`core/cache/CacheManager.h`) | ✓ | `mutable std::mutex m_metaMutex` guards all metadata access (cpp:128/163/181/191/202) | `m_hits`/`m_misses` are `std::atomic` (lock-free) | **SAFE** |
| `TaskScheduler` (`core/scheduler/TaskScheduler.h`) | ✓ | `mutable std::mutex m_graphMtx` guards task-graph state | `s_nextId` is `std::atomic<uint64_t>` | **SAFE** |
| `ImageRepository` (`core/image/ImageRepository.h`) | ✓ | delegates locking to inner `ImageCache` (`pool.mtx`) + `DiskCache` (`s_createMutex` QMutex) — no locking in `ImageRepository.cpp` itself | n/a | **SAFE by delegation** |
| `MemoryTracker` (`core/perf/MemoryTracker.h`) | ✓ | `mutable std::mutex m_peakMtx` guards peak/RSS tracking | frame counters `std::atomic` | **SAFE** |

**Lifecycle / shutdown**: singletons live for process duration; no explicit
`shutdown()` is required because they hold only memory caches and atomics — they
are destroyed at static-exit with no cross-thread ordering hazard (all worker
threads are joined by `TaskScheduler` before process exit via its pool teardown).
No use-after-free or double-free path identified.

**Findings**: none blocking. Two minor notes (non-blocking, deferred):
- `CacheManager` metadata mutex is a single coarse lock; under pathological
  >50k-entry churn it could contend, but measured B6 RSS stayed at 570 MB @1000
  images (M12.2), so no production pressure observed.
- `DiskCache` uses a function-local `static QMutex` (cpp:97) — fine, but a
  `static` Meyers-style wrapper would be more idiomatic than a function-local
  static; cosmetic only.

**Conclusion**: thread-safety posture is sound; no code change required for M12.5.

## 5.3 API documentation skeleton (public core surface)

The four headline APIs get a stable reference page each under `docs/api/`. These
are **interface contracts** (what callers rely on), not implementation docs.
Status: skeleton committed; full prose to be filled as part of the M12.5
follow-up (or post-1.0 doc sprint). Files created:

- `docs/api/ImageRepository.md` — `load()`, `loadAsync()`, `LoadOptions`, cache
  level, cancellation handle.
- `docs/api/Decoder.md` — decoder registry, format support matrix (JPEG/PNG/
  TIFF/BMP/WebP), decode options, error contract.
- `docs/api/Analyzer.md` — `IAnalyzer`, `AnalyzerRegistry` (`registerAnalyzer` /
  `getAnalyzer` / `runAnalyzer`), built-in analyzers (Histogram, RGB Mean, PSNR,
  SSIM, Sharpness, Noise); Qt-free (no QWidget/QPainter dependency).
- `docs/api/RenderEngine.md` — `Renderer` interface, `SoftwareRenderer`,
  `RenderCommand` pipeline, `RenderEngine` facade; notes the sanctioned
  `QPainter` boundary (see 5.1).

See those files for the per-API contract. The analyzer Registry is the most
product-critical (review P1): `core/analyzer/` must expose `registerAnalyzer()` /
`getAnalyzer()` / `runAnalyzer()` and must NOT depend on Qt UI types.
