# M13.3 — Performance Budget Gate (Acceptance)

**Status:** In progress
**Priority:** P0 (user-requested, explicitly ordered #2 in the Product-Beta plan)
**RFC:** `docs/rfc/011-performance-budget.md` (Accepted)
**Freeze note:** Does NOT touch `build.ps1`, `CMakePresets.json`, or `ci.yml`.
Adds a *new, additive* workflow `perf-gate.yml` (like `release.yml`/`nightly.yml`).

## Goal

Turn the existing benchmark harness (`mviewer_bench`) from a *report-only / nightly*
instrument into a **hard CI gate**: any performance regression vs the committed
baseline blocks the PR. Budgets are **data-driven** via
`benchmark/performance_budget.json` (single source of truth), not hardcoded.

## Why now

The review's M13.3 item asked for "Performance Budget Gate → CI 自动检测性能回归".
Prior state (verified from tree):

- `mviewer_bench --enforce` existed and exited non-zero on violation, BUT
  - only 3 budgets were hardcoded (`B2<100ms`, `B8<16ms`, `B9` soak);
  - it ran **only in `nightly.yml` with `|| true`** → never blocked anything;
  - `performance_budget.json` (the user's explicit schema) did not exist.

So the gate was *present in spirit* but **non-binding**. This milestone makes it binding.

## Design

### 1. Data-driven budgets — `benchmark/performance_budget.json`

Mirrors the user's requested schema exactly:

```json
{
  "schema_version": 1,
  "budgets": {
    "open_folder_ms": 500,
    "first_thumbnail_ms": 100,
    "image_switch_ms": 20,
    "zoom_frame_ms": 16,
    "ui_thread_ms": 16,
    "memory_mb": 512
  },
  "scenario_map": { "B1":"open_folder_ms", "B2":"first_thumbnail_ms",
                    "B8":"image_switch_ms", "B9":"memory_mb" },
  "regression": { "baseline_file":"benchmark/perf_baseline.json",
                   "max_regression_pct":10, "fail_on_regression":true }
}
```

`mviewer_bench --enforce --budget <file>` loads it; the hardcoded
`100.0`/`16.0` constants are replaced by `b.first_thumbnail_ms` /
`b.image_switch_ms` / `b.memory_mb`. A missing/invalid file falls back to
defaults + prints a `LOAD-FAILED` warning (fails open, never silently
passes).

### 2. Hard gate — `.github/workflows/perf-gate.yml` (new, additive)

- Triggers on `pull_request` → `master` and `push` → `master`.
- Builds Release, then runs
  `mviewer_bench --enforce --budget benchmark/performance_budget.json`.
- **No `|| true`** — a non-zero exit fails the job, blocking the PR.
- On failure, uploads `benchmark/report/` as an artifact for diagnosis.
- `ci.yml` (Phase-1 mandatory gate) is untouched.

### 3. Local verification — `bench_enforce` ctest

`src/CMakeLists.txt` adds `add_test(NAME bench_enforce COMMAND mviewer_bench
--enforce --budget ${CMAKE_SOURCE_DIR}/benchmark/performance_budget.json)`.
So `.build.ps1 Test` itself cannot go green with a perf regression —
the gate is enforceable locally, not just in CI.

## Acceptance criteria

| # | Criterion | Verification |
|---|-----------|--------------|
| A1 | `benchmark/performance_budget.json` exists with the 6 budgets | `test -f` + JSON parse |
| A2 | `--enforce --budget` uses JSON limits, not hardcoded | code review + `bench_enforce` green |
| A3 | `bench_enforce` ctest FAILS the local suite on violation | run after injecting a bad budget → exit 1 |
| A4 | `perf-gate.yml` fails a PR when a budget is violated | CI run on a deliberately-slow commit (or dry assertion) |
| A5 | `perf-gate.yml` does NOT modify frozen `ci.yml` | diff shows only a new file |
| A6 | Full `.\build.ps1 Test` stays green (34/34) | background run |

## Out of scope (per freeze + user "no big refactor")

- No change to `Benchmark`/`Scenario` math, `CacheManager`, `ImageRepository`,
  `Scheduler`, `Domain`, `Core` layering.
- GPU tile pipeline (M13.2) deferred per user's explicit "先不要碰 GPU".
- Baseline auto-capture/regression-diff is a follow-up; this milestone makes
  the **threshold gate** real. `perf_baseline.json` is the documented next step.
